### BUG-1: Vistas de disponibilidad y habitaciones no funcionan (CRÍTICO)

**Causa raíz**: Desajuste de contrato entre backend y frontend en `GET /api/room-types`.

| Capa     | Comportamiento real                                                                                   |
|----------|-------------------------------------------------------------------------------------------------------|
| Backend  | `RoomTypeController.list()` devuelve `PageDto<RoomTypeDto>` → `{ content: [...], page, size, totalElements, totalPages }` |
| Frontend | `roomTypesApi.list()` tipa la respuesta como `RoomTypeDto[]` (array plano)                           |
| MSW mock | `http.get('/room-types', () => HttpResponse.json(roomTypes))` devuelve array plano                   |

**Impacto**: `useRoomTypes({})` devuelve un objeto `PageDto` en runtime, pero el
código TypeScript lo trata como array. `AvailabilityPage` y `RoomsPage` llaman
`roomTypesQ.data?.filter(...)` / `.map(...)` sobre el objeto →
`TypeError: .filter is not a function` → la página crashea.

**Por qué las pruebas no lo detectan**: MSW mockea `/room-types` como array
plano, coincidiendo con lo que espera el frontend, pero no con el backend real.

**Evidencia (curl al backend real)**:
```
GET /api/room-types → {"content":[{"id":1,"name":"Sencilla",...}],...}  (objeto PageDto)
MSW mock            → [{"id":1,"name":"Sencilla",...},...]               (array plano)
```

**Solución**: El frontend `roomTypesApi.list()` debe manejar la respuesta
paginada y extraer `.content`, o el backend debe ofrecer un endpoint que
devuelva array plano. **Decisión**: alinear el frontend para manejar `PageDto`
y extraer `.content` (el backend ya paginado es correcto para consistencia con
otros listados; los dropdowns usan `size=100` para traer todo).

### BUG-2: BigDecimal serializado como string

**Causa raíz**: `JacksonConfig.BigDecimalScaleSerializer` usa
`gen.writeString(value.setScale(2, ...).toPlainString())` → serializa como
string `"120.00"` en vez de número `120.00`.

**Evidencia**:
```json
{"basePrice":"120.00","totalAmount":"240.00"}  // string, no number
```

**Impacto**: Los tipos TypeScript del frontend declaran `basePrice: number`
pero runtime recibe string. `Intl.NumberFormat.format()` coerce strings, pero
es inconsistente y puede causar bugs sutiles (comparaciones, cálculos).

**Solución**: Cambiar el serializer a `gen.writeNumber(value.setScale(2, ...))`
para serializar como número con 2 decimales.

### BUG-3: Moneda mostrada en EUR, debe ser MXN

**Causa raíz**: `frontend/src/utils/format.ts` → `formatCurrency()` usa
`Intl.NumberFormat('es-ES', { currency: 'EUR' })`.

**Solución**: Cambiar a `Intl.NumberFormat('es-MX', { currency: 'MXN' })`.
El backend debe usar código de moneda `MXN` donde se mencione moneda.
Los importes existentes (semilla y históricos) se tratan como MXN; no se
convierten. Solo se ajustan cantidades de demostración si no son realistas
para MXN.

### BUG-4: `active: null` en respuesta de login

**Causa raíz**: `AuthService.login()` crea `UserDto` con constructor de 4
args (sin `active`), mientras que `me()` usa constructor de 5 args.

**Solución**: Usar constructor de 5 args en `login()` también.

### BUG-5: Sin refresh tokens

**Estado actual**: Solo access token JWT (1h), sin refresh. Al expirar,
usuario debe re-login.

**Solución**: Implementar refresh tokens con rotación, almacenamiento de
hash, cookies HttpOnly, y detección de reutilización.

## 3. Contratos para los agentes

### 3.1 Tabla `refresh_tokens` (Database + Backend)

```sql
CREATE TABLE refresh_tokens (
    id              BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id         BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token_hash      TEXT NOT NULL UNIQUE,          -- SHA-256 hash, nunca el token original
    jti             TEXT NOT NULL UNIQUE,          -- JWT ID único
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    expires_at      TIMESTAMPTZ NOT NULL,
    revoked_at      TIMESTAMPTZ,                    -- NULL = activo; fecha = revocado
    replaced_by_jti TEXT,                           -- JTI del token que lo reemplazó (rotación)
    device_info     TEXT,                           -- User-Agent / info de sesión (opcional)
    CONSTRAINT refresh_tokens_check_expiry CHECK (expires_at > created_at)
);
CREATE INDEX idx_refresh_tokens_user_id    ON refresh_tokens (user_id);
CREATE INDEX idx_refresh_tokens_jti        ON refresh_tokens (jti);
CREATE INDEX idx_refresh_tokens_token_hash ON refresh_tokens (token_hash);
CREATE INDEX idx_refresh_tokens_expires_at ON refresh_tokens (expires_at);
```

### 3.2 Endpoints de auth (Backend + Frontend)

| Método | Path               | Auth | Descripción                                      |
|--------|--------------------|------|--------------------------------------------------|
| POST   | `/api/auth/login`  | No   | Devuelve access token + set-cookie refresh       |
| POST   | `/api/auth/refresh`| Cookie refresh | Rota refresh token, devuelve nuevo access token + nueva cookie |
| POST   | `/api/auth/logout` | Access token | Revoca refresh token, limpia cookie             |
| GET    | `/api/auth/me`     | Access token | Perfil del usuario                              |

**Cookie del refresh token**:
- Name: `hotel_refresh`
- HttpOnly: true
- SameSite: Lax (dev) / Strict (prod configurable)
- Secure: configurable via `app.refresh-token.secure` (false en dev, true en prod)
- Path: `/api/auth`
- Max-Age: duración del refresh token

**Login response (actualizado)**:
```jsonc
{
  "token": "string",         // access token (corto, 15 min)
  "type": "Bearer",
  "expiresIn": 900,          // 15 minutos
  "user": { "id": 1, "email": "string", "fullName": "string", "role": "ADMIN", "active": true }
}
// + Set-Cookie: hotel_refresh=<refresh-token>; HttpOnly; Path=/api/auth; SameSite=Lax; Max-Age=604800
```

**Refresh request**: no body; lee cookie `hotel_refresh`.
**Refresh response**:
```jsonc
{ "token": "string", "type": "Bearer", "expiresIn": 900 }
// + Set-Cookie: hotel_refresh=<new-refresh-token>; HttpOnly; Path=/api/auth; ...
```

**Rotación**: cada uso del refresh token genera uno nuevo, revoca el anterior
(`revoked_at` + `replaced_by_jti`), y devuelve la nueva cookie. Si se reutiliza
un token revocado → revocar toda la cadena (detección de reutilización) → 401.

**Configuración**:
- `app.jwt.access-token-expiration-seconds`: 900 (15 min)
- `app.refresh-token.expiration-seconds`: 604800 (7 días)
- `app.refresh-token.secure`: false (dev) / true (prod)

**CORS**: ya tiene `allowCredentials(true)`. Verificar que el frontend envía
`withCredentials: true` en axios.
