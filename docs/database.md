## 1. Convenciones

| Aspecto              | DecisiГіn                                                                 |
|----------------------|--------------------------------------------------------------------------|
| IDs                  | `BIGINT GENERATED BY DEFAULT AS IDENTITY` (PK).                          |
| Dinero               | `NUMERIC(12,2)` (moneda implГӯcita **MXN**; `BigDecimal` en Java).        |
| Fechas calendario    | `DATE` (`check_in` / `check_out`).                                       |
| Marcas de auditorГӯa  | `TIMESTAMPTZ` (UTC), `DEFAULT now()`.                                    |
| Texto libre          | `TEXT`.                                                                  |
| Amenities / metadata | `JSONB` (`'[]'` / `'{}'` por defecto).                                   |
| Enums                | `CHECK (col IN (...))` (no tabla catГЎlogo).                              |
| `updated_at`         | **Sin trigger**; lo actualiza el backend en cada servicio (JPA `@PreUpdate`). |
| Soft deletes         | No. Se usan estados (`active`, `status`) y `ON DELETE` estricto.         |

> **Moneda:** todas las cantidades `NUMERIC(12,2)` se interpretan como **MXN**
> (peso mexicano). HistГіricamente el esquema se documentГі como EUR; a partir de
> `V4__adjust_seed_prices_mxn.sql` los importes semilla son realistas para MXN.
> Los importes preexistentes se tratan como MXN (no hay conversiГіn de moneda:
> solo se ajustaron cantidades de demostraciГіn). El cГіdigo de moneda para
> presentation / OpenAPI es `MXN`. Ver BUG-3 en `fix-plan.md`.

> Se elige `GENERATED BY DEFAULT AS IDENTITY` (en vez de `ALWAYS`) para permitir
> inserts semilla con `id` explГӯcito sin `OVERRIDING SYSTEM VALUE`. Tras la
> semilla, `V2` resincroniza las secuencias con `setval(pg_get_serial_sequence(...))`.

### `refresh_tokens`
Persistencia de refresh tokens para el flujo de autenticaciГіn con rotaciГіn
(BUG-5 / `fix-plan.md` В§3.1). Creada en `V3__refresh_tokens.sql`.

| Columna           | Tipo           | Nulidad | Restricciones / FK                                  |
|-------------------|----------------|---------|-----------------------------------------------------|
| `id`              | BIGINT         | NOT NULL| PK, identity                                        |
| `user_id`         | BIGINT         | NOT NULL| FK вҶ’ `users(id)` ON DELETE CASCADE                  |
| `token_hash`      | TEXT           | NOT NULL| UNIQUE; SHA-256 del token (nunca el token original) |
| `jti`             | TEXT           | NOT NULL| UNIQUE; JWT ID del token emitido                    |
| `created_at`      | TIMESTAMPTZ    | NOT NULL| DEFAULT now()                                       |
| `expires_at`      | TIMESTAMPTZ    | NOT NULL| CHECK (`expires_at > created_at`)                   |
| `revoked_at`      | TIMESTAMPTZ    | NULL    | NULL = activo; fecha = revocado (rotaciГіn/logout)   |
| `replaced_by_jti` | TEXT           | NULL    | JTI del token que lo reemplazГі al rotar             |
| `device_info`     | TEXT           | NULL    | User-Agent / info de sesiГіn (opcional)              |

RestricciГіn adicional: `refresh_tokens_check_expiry`
`CHECK (expires_at > created_at)`.

Notas de diseГұo:
- **No se guarda el token original**: solo `token_hash` (SHA-256) y `jti`. El
  backend compara el hash del token recibido contra `token_hash`.
- `ON DELETE CASCADE` desde `users`: borrar un usuario elimina sus refresh
  tokens (coherente con soft-delete por `active`; el borrado fГӯsico de usuarios
  no es el flujo normal).
- `revoked_at IS NULL` identifica un token activo; `revoked_at` con fecha
  indica rotaciГіn o logout. `replaced_by_jti` enlaza la cadena de rotaciГіn.
- La limpieza de tokens expirados/revocados la realiza el backend (job de
  mantenimiento o `DELETE WHERE expires_at < now()`); la BD no la automatiza.

## 4. PrevenciГіn de solapamiento (RN-2)

### 4.1 Mecanismo a nivel de BD

`reservation_rooms` lleva una **columna generada STORED**:

```sql
overlap_range daterange NOT NULL
    GENERATED ALWAYS AS (daterange(check_in, check_out, '[)')) STORED
```

- `daterange(..., '[)')` вҶ’ lГӯmite inferior **inclusivo**, superior **exclusivo**
  (la noche de `check_out` **no** cuenta como ocupada, coherente con RN-1).
- `daterange()` sobre `DATE` es **inmutable**, requisito para
  `GENERATED вҖҰ STORED` en PostgreSQL 16.

Y una **restricciГіn de exclusiГіn GiST**:

```sql
ALTER TABLE reservation_rooms
    ADD CONSTRAINT reservation_rooms_no_overlap
    EXCLUDE USING gist (room_id WITH =, overlap_range WITH &&);
```

- `room_id WITH =` вҶ’ impide misma habitaciГіn; necesita el opclass GiST de
  **`btree_gist`** para `BIGINT` (de ahГӯ `CREATE EXTENSION IF NOT EXISTS btree_gist;`
  al inicio de `V1`).
- `overlap_range WITH &&` вҶ’ impide rangos solapados; `range_gist` viene incluido
  en PostgreSQL.

Resultado: la BD **rechaza** (`SQLSTATE 23P01`, violaciГіn de exclusiГіn) cualquier
`INSERT`/`UPDATE` que produzca dos filas con el mismo `room_id` y rangos que se
solapen. Esto es una **red de seguridad** ante concurrencia, complementaria al
bloqueo pesimista/optimista de la aplicaciГіn (RN-9).

### 4.2 Por quГ© la restricciГіn **no es suficiente** por sГӯ sola

La exclusiГіn opera sobre **todas** las filas de `reservation_rooms`, sin conocer
el `status` de la `reservations` padre. Una reserva `CANCELLED` o `NO_SHOW` no
debe bloquear disponibilidad (RN-2, RN-4). Por tanto el modelo se complementa con
**reglas de aplicaciГіn**:

1. **Solo se mantienen filas en `reservation_rooms` para reservas activas**
   (estado вү  `CANCELLED` y вү  `NO_SHOW`).
2. **Al cancelar** una reserva (o marcar `NO_SHOW`), el backend debe **`DELETE`
   de `reservation_rooms`** de esa reserva вҶ’ libera la habitaciГіn (RN-4).
   Gracias a `ON DELETE CASCADE` en `reservation_rooms.reservation_id`, borrar la
   reserva tambiГ©n las elimina, pero el flujo normal es cancelar (no borrar) la
   reserva, asГӯ que el backend debe eliminar las asignaciones explГӯcitamente.
3. **AsignaciГіn / reasignaciГіn** (`assign-room`, `check-in` con asignaciГіn
   automГЎtica): el servicio valida dentro de la **misma transacciГіn**, con
   bloqueo (`SELECT вҖҰ FOR UPDATE` sobre `rooms` o `@Version`) y comprobando
   que no exista `reservation_rooms` solapante para esa habitaciГіn entre
   reservas activas. La violaciГіn de la exclusiГіn GiST se traduce a
   `409 RESERVATION_OVERLAP` (RN-9, criterio 5).
4. **Consulta de disponibilidad** (`availability`): habitaciones cuyo tipo soporta
   el total de huГ©spedes (`adults + children <= max_capacity`) y que **no** tengan
   ninguna fila en `reservation_rooms` (de reserva activa) con
   `overlap_range && daterange(:checkIn, :checkOut, '[)')`.

> En resumen: la **exclusiГіn GiST** garantiza la invariant a nivel de filas
> presentes; las **reglas de borrado al cancelar** garantizan que las filas
> presentes correspondan solo a reservas activas. Juntas implementan RN-2.

### 4.3 ExtensiГіn `btree_gist`

`V1__init_schema.sql` ejecuta `CREATE EXTENSION IF NOT EXISTS btree_gist;`.

- En **dev** (`postgres:16-alpine`, `POSTGRES_USER=hotel`): el rol es
  **superusuario** en la imagen oficial вҶ’ puede crear la extensiГіn.
- En **prod**: si el rol de la app no es superusuario, crea la extensiГіn
  previamente con un superusuario:
  `CREATE EXTENSION IF NOT EXISTS btree_gist;` (idempotente; segura en
  transacciГіn, asГӯ que tambiГ©n puede ir en un script de init del contenedor).
- `btree_gist` viene en el paquete contrib de PostgreSQL 16, incluido en la
  imagen `alpine` usada por `compose.yaml`.

## 7. Contrato para el BACKEND agent (obligatorio)

### 7.1 `DataInitializer` para `BCRYPT_PENDING`

`V2__seed_data.sql` inserta los usuarios con `password_hash = 'BCRYPT_PENDING'`
(centinela) en lugar de un hash BCrypt codificado en SQL plano, para **no
enviar hashes invГЎlidos** (un hash BCrypt vГЎlido debe generarse con
`BCryptPasswordEncoder` en JVM; escribirlo a mano en SQL es frГЎgil).

El backend **debe** implementar un `DataInitializer`
(`CommandLineRunner` / `ApplicationRunner` / bean con `@PostConstruct`) que se
ejecute **despuГ©s** de Flyway y haga:

```java
@Component
@RequiredArgsConstructor
class DataInitializer {
    private final UserRepository userRepository;
    private final PasswordEncoder passwordEncoder; // BCryptPasswordEncoder coste 10

    private static final Map<String,String> SEED_CREDENTIALS = Map.of(
        "admin@hotel.test",     "admin123",
        "recepcion@hotel.test", "recepcion123"
    );

    @PostConstruct
    void patchSeedPasswords() {
        for (User u : userRepository.findAll()) {
            if ("BCRYPT_PENDING".equals(u.getPasswordHash())) {
                String plain = SEED_CREDENTIALS.get(u.getEmail());
                if (plain != null) {
                    u.setPasswordHash(passwordEncoder.encode(plain));
                    userRepository.save(u);
                }
            }
        }
    }
}
```

Equivalente SQL: `UPDATE users SET password_hash = :bcrypt
WHERE password_hash = 'BCRYPT_PENDING';` (usando `BCryptPasswordEncoder.encode`).

El `DataInitializer` debe ser **idempotente** (solo actualiza las filas con
centinela) y seguro de ejecutar en cada arranque. Idealmente debe logs informativo
de cuГЎntas filas parchea.

### 7.2 JPA вҶ” SQL: `updated_at`

No hay triggers `BEFORE UPDATE` sobre `updated_at`. El backend debe actualizar
`updated_at` en cada `save`/`update` (p. ej. `@PreUpdate` en la entidad o en el
servicio). Esto coincide con `SPRING_JPA_HIBERNATE_DDL_AUTO=validate`
(`.env.example`): Hibernate **no** crea ni altera tablas; Flyway es la autoridad.
Las entidades JPA deben mapear exactamente columnas y tipos de aquГӯ (validaciГіn
fallarГЎ si hay desviaciГіn).

## 9. Refresh tokens: persistencia y rotaciГіn

Implementa BUG-5 / `fix-plan.md` В§3.1. La tabla `refresh_tokens` (migraciГіn
`V3`) es el almacГ©n de la capa de persistencia para el flujo de autenticaciГіn con
**rotaciГіn** y **detecciГіn de reutilizaciГіn**.

### 9.1 Modelo de almacenamiento

- **Nunca se persiste el token original.** Se guarda `token_hash` = SHA-256 del
  refresh token emitido, y `jti` = JWT ID Гәnico por token. La verificaciГіn
  consiste en hashear el token recibido de la cookie y buscar por `token_hash`.
- `user_id` referencia `users(id)` con `ON DELETE CASCADE`: eliminar un usuario
  borra sus refresh tokens (el borrado fГӯsico de usuarios no es el flujo normal;
  la desactivaciГіn se hace con `active = false`).
- `expires_at` obligatorio y `CHECK (expires_at > created_at)`: el token nace
  vГЎlido. La duraciГіn la fija el backend (`app.refresh-token.expiration-seconds`,
  por defecto 604800 = 7 dГӯas).
- `revoked_at` nullable: `NULL` = activo; fecha = revocado (por rotaciГіn o por
  logout). `replaced_by_jti` apunta al `jti` del token que lo reemplazГі, formando
  la **cadena de rotaciГіn**.
- `device_info` opcional (User-Agent / info de sesiГіn) para auditorГӯa y posibles
  listados de sesiones activas.

### 9.2 RotaciГіn (cada uso)

Al procesar `POST /api/auth/refresh`:

1. Leer cookie `hotel_refresh`, calcular `token_hash = sha256(token)`, buscar la
   fila por `token_hash`.
2. Validar: existe, `revoked_at IS NULL`, `expires_at > now()`, `user.active`.
3. Generar nuevo refresh token con `jti'` nuevo, persistirlo como fila nueva.
4. **Revocar el anterior**: `UPDATE refresh_tokens SET revoked_at = now(),
   replaced_by_jti = :newJti WHERE id = :oldId`.
5. Emitir nuevo access token + nueva cookie `hotel_refresh`.

Todo en la **misma transacciГіn**.

### 9.3 DetecciГіn de reutilizaciГіn

Si llega un refresh token cuyo `token_hash` corresponde a una fila con
`revoked_at IS NOT NULL` (ya fue rotado): alguien estГЎ reutilizando un token
viejo (posible robo de cookie). AcciГіn:

- Revocar **toda la cadena** (la fila presentada y las que ella reemplazГі / la
  reemplazaron) y, segГәn polГӯtica, todos los refresh tokens activos del usuario.
- Devolver **`401`** y forzar re-login.

### 9.4 Logout

`POST /api/auth/logout`: revoca el refresh token de la cookie
(`UPDATE ... SET revoked_at = now() WHERE token_hash = :hash AND revoked_at IS
NULL`) y borra la cookie del cliente. No se elimina la fila (se conserva para
auditorГӯa / detecciГіn de reutilizaciГіn).

### 9.5 Purga

La BD no automatiza la limpieza. El backend (job de mantenimiento) puede ejecutar
periГіdicamente:

```sql
DELETE FROM refresh_tokens
 WHERE expires_at < now()
   AND revoked_at IS NOT NULL;
```

Los tokens expirados no activos pueden borrarse con seguridad; conservar los
revocados mientras puedan ser presentados de nuevo ayuda a la detecciГіn de
reutilizaciГіn. El Гӯndice `idx_refresh_tokens_expires_at` acelera esta operaciГіn.

### 9.6 Contrato con el backend

- La entidad JPA `RefreshToken` mapea exactamente las columnas de la tabla
  (`Long id`, `Long userId` (o `User user`), `String tokenHash`, `String jti`,
  `Instant createdAt`, `Instant expiresAt`, `Instant revokedAt`,
  `String replacedByJti`, `String deviceInfo`).
- `token_hash` y `jti` son `UNIQUE`: una violaciГіn (`SQLSTATE 23505`) debe
  traducirse a `409 CONFLICT` / reintentar con un `jti` nuevo (colisiГіn de JTI
  improbable pero posible).
- No hay `updated_at` en esta tabla: los cambios de estado se registran con
  `revoked_at` (audit timestamp), no con un `updated_at` genГ©rico.
- El hash debe calcularse con SHA-256 sobre el token **completo** (string del
  JWT), en hex o base64 consistente; el backend es el dueГұo de la codificaciГіn.
