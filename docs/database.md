## 1. Convenciones

| Aspecto              | Decisión                                                                 |
|----------------------|--------------------------------------------------------------------------|
| IDs                  | `BIGINT GENERATED BY DEFAULT AS IDENTITY` (PK).                          |
| Dinero               | `NUMERIC(12,2)` (moneda implícita **MXN**; `BigDecimal` en Java).        |
| Fechas calendario    | `DATE` (`check_in` / `check_out`).                                       |
| Marcas de auditoría  | `TIMESTAMPTZ` (UTC), `DEFAULT now()`.                                    |
| Texto libre          | `TEXT`.                                                                  |
| Amenities / metadata | `JSONB` (`'[]'` / `'{}'` por defecto).                                   |
| Enums                | `CHECK (col IN (...))` (no tabla catálogo).                              |
| `updated_at`         | **Sin trigger**; lo actualiza el backend en cada servicio (JPA `@PreUpdate`). |
| Soft deletes         | No. Se usan estados (`active`, `status`) y `ON DELETE` estricto.         |

> **Moneda:** todas las cantidades `NUMERIC(12,2)` se interpretan como **MXN**
> (peso mexicano). Históricamente el esquema se documentó como EUR; a partir de
> `V4__adjust_seed_prices_mxn.sql` los importes semilla son realistas para MXN.
> Los importes preexistentes se tratan como MXN (no hay conversión de moneda:
> solo se ajustaron cantidades de demostración). El código de moneda para
> presentation / OpenAPI es `MXN`. Ver BUG-3 en `fix-plan.md`.

> Se elige `GENERATED BY DEFAULT AS IDENTITY` (en vez de `ALWAYS`) para permitir
> inserts semilla con `id` explícito sin `OVERRIDING SYSTEM VALUE`. Tras la
> semilla, `V2` resincroniza las secuencias con `setval(pg_get_serial_sequence(...))`.

### `refresh_tokens`
Persistencia de refresh tokens para el flujo de autenticación con rotación
(BUG-5 / `fix-plan.md` §3.1). Creada en `V3__refresh_tokens.sql`.

| Columna           | Tipo           | Nulidad | Restricciones / FK                                  |
|-------------------|----------------|---------|-----------------------------------------------------|
| `id`              | BIGINT         | NOT NULL| PK, identity                                        |
| `user_id`         | BIGINT         | NOT NULL| FK → `users(id)` ON DELETE CASCADE                  |
| `token_hash`      | TEXT           | NOT NULL| UNIQUE; SHA-256 del token (nunca el token original) |
| `jti`             | TEXT           | NOT NULL| UNIQUE; JWT ID del token emitido                    |
| `created_at`      | TIMESTAMPTZ    | NOT NULL| DEFAULT now()                                       |
| `expires_at`      | TIMESTAMPTZ    | NOT NULL| CHECK (`expires_at > created_at`)                   |
| `revoked_at`      | TIMESTAMPTZ    | NULL    | NULL = activo; fecha = revocado (rotación/logout)   |
| `replaced_by_jti` | TEXT           | NULL    | JTI del token que lo reemplazó al rotar             |
| `device_info`     | TEXT           | NULL    | User-Agent / info de sesión (opcional)              |

Restricción adicional: `refresh_tokens_check_expiry`
`CHECK (expires_at > created_at)`.

Notas de diseño:
- **No se guarda el token original**: solo `token_hash` (SHA-256) y `jti`. El
  backend compara el hash del token recibido contra `token_hash`.
- `ON DELETE CASCADE` desde `users`: borrar un usuario elimina sus refresh
  tokens (coherente con soft-delete por `active`; el borrado físico de usuarios
  no es el flujo normal).
- `revoked_at IS NULL` identifica un token activo; `revoked_at` con fecha
  indica rotación o logout. `replaced_by_jti` enlaza la cadena de rotación.
- La limpieza de tokens expirados/revocados la realiza el backend (job de
  mantenimiento o `DELETE WHERE expires_at < now()`); la BD no la automatiza.

### `reservation_nightly_rates` (V6, ER-9)

`check_out` (exclusivo). `total` se valida con CHECK aritmético.

| `rate_plan_id`         | BIGINT         | NOT NULL| FK → `rate_plans(id)` ON DELETE RESTRICT                       |

| `base_rate`            | NUMERIC(12,2)  | NOT NULL| CHECK `>= 0`                                                   |
| `extra_person_charge`  | NUMERIC(12,2)  | NOT NULL| DEFAULT 0; CHECK `>= 0`                                        |
| `discount_amount`      | NUMERIC(12,2)  | NOT NULL| DEFAULT 0; CHECK `>= 0`                                        |

## 4. Prevención de solapamiento (RN-2)

### 4.1 Mecanismo a nivel de BD

`reservation_rooms` lleva una **columna generada STORED**:

```sql
overlap_range daterange NOT NULL
    GENERATED ALWAYS AS (daterange(check_in, check_out, '[)')) STORED
```

- `daterange(..., '[)')` → límite inferior **inclusivo**, superior **exclusivo**
  (la noche de `check_out` **no** cuenta como ocupada, coherente con RN-1).
- `daterange()` sobre `DATE` es **inmutable**, requisito para
  `GENERATED … STORED` en PostgreSQL 16.

Y una **restricción de exclusión GiST**:

```sql
ALTER TABLE reservation_rooms
    ADD CONSTRAINT reservation_rooms_no_overlap
    EXCLUDE USING gist (room_id WITH =, overlap_range WITH &&);
```

- `room_id WITH =` → impide misma habitación; necesita el opclass GiST de
  **`btree_gist`** para `BIGINT` (de ahí `CREATE EXTENSION IF NOT EXISTS btree_gist;`
  al inicio de `V1`).
- `overlap_range WITH &&` → impide rangos solapados; `range_gist` viene incluido
  en PostgreSQL.

Resultado: la BD **rechaza** (`SQLSTATE 23P01`, violación de exclusión) cualquier
`INSERT`/`UPDATE` que produzca dos filas con el mismo `room_id` y rangos que se
solapen. Esto es una **red de seguridad** ante concurrencia, complementaria al
bloqueo pesimista/optimista de la aplicación (RN-9).

### 4.2 Por qué la restricción **no es suficiente** por sí sola

La exclusión opera sobre **todas** las filas de `reservation_rooms`, sin conocer
el `status` de la `reservations` padre. Una reserva `CANCELLED` o `NO_SHOW` no
debe bloquear disponibilidad (RN-2, RN-4). Por tanto el modelo se complementa con
**reglas de aplicación**:

1. **Solo se mantienen filas en `reservation_rooms` para reservas activas**
   (estado ≠ `CANCELLED` y ≠ `NO_SHOW`).
2. **Al cancelar** una reserva (o marcar `NO_SHOW`), el backend debe **`DELETE`
   de `reservation_rooms`** de esa reserva → libera la habitación (RN-4).
   Gracias a `ON DELETE CASCADE` en `reservation_rooms.reservation_id`, borrar la
   reserva también las elimina, pero el flujo normal es cancelar (no borrar) la
   reserva, así que el backend debe eliminar las asignaciones explícitamente.
3. **Asignación / reasignación** (`assign-room`, `check-in` con asignación
   automática): el servicio valida dentro de la **misma transacción**, con
   bloqueo (`SELECT … FOR UPDATE` sobre `rooms` o `@Version`) y comprobando
   que no exista `reservation_rooms` solapante para esa habitación entre
   reservas activas. La violación de la exclusión GiST se traduce a
   `409 RESERVATION_OVERLAP` (RN-9, criterio 5).
4. **Consulta de disponibilidad** (`availability`): habitaciones cuyo tipo soporta
   el total de huéspedes (`adults + children <= max_capacity`) y que **no** tengan
   ninguna fila en `reservation_rooms` (de reserva activa) con
   `overlap_range && daterange(:checkIn, :checkOut, '[)')`.

> En resumen: la **exclusión GiST** garantiza la invariant a nivel de filas
> presentes; las **reglas de borrado al cancelar** garantizan que las filas
> presentes correspondan solo a reservas activas. Juntas implementan RN-2.

### 4.3 Extensión `btree_gist`

`V1__init_schema.sql` ejecuta `CREATE EXTENSION IF NOT EXISTS btree_gist;`.

- En **dev** (`postgres:16-alpine`, `POSTGRES_USER=hotel`): el rol es
  **superusuario** en la imagen oficial → puede crear la extensión.
- En **prod**: si el rol de la app no es superusuario, crea la extensión
  previamente con un superusuario:
  `CREATE EXTENSION IF NOT EXISTS btree_gist;` (idempotente; segura en
  transacción, así que también puede ir en un script de init del contenedor).
- `btree_gist` viene en el paquete contrib de PostgreSQL 16, incluido en la
  imagen `alpine` usada por `compose.yaml`.

## 6. Cuentas de desarrollo

No existen credenciales semilla versionadas. Las cuentas se proporcionan desde
el `.env` ignorado o un gestor de secretos y se crean solo cuando
`BOOTSTRAP_USERS_ENABLED=true`. Después del primer arranque se debe volver a
desactivar el bootstrap.

## 7. Contrato para el BACKEND agent (obligatorio)

### 7.1 `DataInitializer` con secretos de entorno

El `DataInitializer` es opt-in. Si está habilitado, exige email y contraseña
para cada rol, valida la política de contraseña, genera BCrypt y crea únicamente
cuentas ausentes. No modifica contraseñas existentes ni contiene literales de
acceso. La V14 desactiva cuentas históricas y revoca sus sesiones.

### 7.2 JPA ↔ SQL: `updated_at`

No hay triggers `BEFORE UPDATE` sobre `updated_at`. El backend debe actualizar
`updated_at` en cada `save`/`update` (p. ej. `@PreUpdate` en la entidad o en el
servicio). Esto coincide con `SPRING_JPA_HIBERNATE_DDL_AUTO=validate`
(`.env.example`): Hibernate **no** crea ni altera tablas; Flyway es la autoridad.
Las entidades JPA deben mapear exactamente columnas y tipos de aquí (validación
fallará si hay desviación).

### 7.3 Mapeos sugeridos (resumen)

| SQL                        | Java / JPA                                   |
|----------------------------|----------------------------------------------|
| `BIGINT IDENTITY`          | `Long` `@GeneratedValue(strategy = IDENTITY)`|
| `NUMERIC(12,2)`            | `BigDecimal` (`@Column(scale = 2)`)          |
| `DATE`                     | `LocalDate`                                  |
| `TIMESTAMPTZ`              | `Instant` (recomendado) o `OffsetDateTime`   |
| `JSONB` (amenities)        | `List<String>` con `JsonAttributeConverter` / Hibernate JSON |
| `JSONB` (audit metadata)   | `Map<String,Object>` con converter JSON      |

> La entidad `ReservationRoom` **no** debe intentar escribir `overlap_range`
> (es `GENERATED ALWAYS`). Tampoco debe incluirse en inserts nativos. El backend

## 9. Refresh tokens: persistencia y rotación

Implementa BUG-5 / `fix-plan.md` §3.1. La tabla `refresh_tokens` (migración
`V3`) es el almacén de la capa de persistencia para el flujo de autenticación con
**rotación** y **detección de reutilización**.

### 9.1 Modelo de almacenamiento

- **Nunca se persiste el token original.** Se guarda `token_hash` = SHA-256 del
  refresh token emitido, y `jti` = JWT ID único por token. La verificación
  consiste en hashear el token recibido de la cookie y buscar por `token_hash`.
- `user_id` referencia `users(id)` con `ON DELETE CASCADE`: eliminar un usuario
  borra sus refresh tokens (el borrado físico de usuarios no es el flujo normal;
  la desactivación se hace con `active = false`).
- `expires_at` obligatorio y `CHECK (expires_at > created_at)`: el token nace
  válido. La duración la fija el backend (`app.refresh-token.expiration-seconds`,
  por defecto 604800 = 7 días).
- `revoked_at` nullable: `NULL` = activo; fecha = revocado (por rotación o por
  logout). `replaced_by_jti` apunta al `jti` del token que lo reemplazó, formando
  la **cadena de rotación**.
- `device_info` opcional (User-Agent / info de sesión) para auditoría y posibles
  listados de sesiones activas.

### 9.2 Rotación (cada uso)

Al procesar `POST /api/auth/refresh`:

1. Leer cookie `hotel_refresh`, calcular `token_hash = sha256(token)`, buscar la
   fila por `token_hash`.
2. Validar: existe, `revoked_at IS NULL`, `expires_at > now()`, `user.active`.
3. Generar nuevo refresh token con `jti'` nuevo, persistirlo como fila nueva.
4. **Revocar el anterior**: `UPDATE refresh_tokens SET revoked_at = now(),
   replaced_by_jti = :newJti WHERE id = :oldId`.
5. Emitir nuevo access token + nueva cookie `hotel_refresh`.

Todo en la **misma transacción**.

### 9.3 Detección de reutilización

Si llega un refresh token cuyo `token_hash` corresponde a una fila con
`revoked_at IS NOT NULL` (ya fue rotado): alguien está reutilizando un token
viejo (posible robo de cookie). Acción:

- Revocar **toda la cadena** (la fila presentada y las que ella reemplazó / la
  reemplazaron) y, según política, todos los refresh tokens activos del usuario.
- Devolver **`401`** y forzar re-login.

### 9.4 Logout

`POST /api/auth/logout`: revoca el refresh token de la cookie
(`UPDATE ... SET revoked_at = now() WHERE token_hash = :hash AND revoked_at IS
NULL`) y borra la cookie del cliente. No se elimina la fila (se conserva para
auditoría / detección de reutilización).

### 9.5 Purga

La BD no automatiza la limpieza. El backend (job de mantenimiento) puede ejecutar
periódicamente:

```sql
DELETE FROM refresh_tokens
 WHERE expires_at < now()
   AND revoked_at IS NOT NULL;
```

Los tokens expirados no activos pueden borrarse con seguridad; conservar los
revocados mientras puedan ser presentados de nuevo ayuda a la detección de
reutilización. El índice `idx_refresh_tokens_expires_at` acelera esta operación.

### 9.6 Contrato con el backend

- La entidad JPA `RefreshToken` mapea exactamente las columnas de la tabla
  (`Long id`, `Long userId` (o `User user`), `String tokenHash`, `String jti`,
  `Instant createdAt`, `Instant expiresAt`, `Instant revokedAt`,
  `String replacedByJti`, `String deviceInfo`).
- `token_hash` y `jti` son `UNIQUE`: una violación (`SQLSTATE 23505`) debe
  traducirse a `409 CONFLICT` / reintentar con un `jti` nuevo (colisión de JTI
  improbable pero posible).
- No hay `updated_at` en esta tabla: los cambios de estado se registran con
  `revoked_at` (audit timestamp), no con un `updated_at` genérico.
- El hash debe calcularse con SHA-256 sobre el token **completo** (string del
  JWT), en hex o base64 consistente; el backend es el dueño de la codificación.

## 10. Motor tarifario (V5, ER-1..ER-6.2)

Implementa la sección 1 de `hotel-rules.md`. Tablas: `rate_plans`,
`seasonal_rates`, `daily_rate_overrides`, `promotion_rules`, `closed_dates`,
`taxes_and_fees`.

### 10.1 Precedencia de tarifas (ER-7)

Para una noche dada, el motor tarifario resuelve el precio en este orden (de
mayor a menor prioridad):

1. `daily_rate_overrides` para esa fecha + `room_type_id` + `rate_plan_id` (o
   `rate_plan_id IS NULL` = aplica a cualquier plan).
2. `seasonal_rates` del plan tarifario vigentes en esa fecha
   (`overlap_range && daterange(night, night+1, '[)')`).
3. `weekday_rates` del plan tarifario (índice 0=lun … 6=dom).
4. `room_types.base_price` (fallback).

### 10.2 Inmutabilidad del snapshot (ER-9)

Al confirmar una reserva (PENDING → CONFIRMED), el backend crea una fila
`reservation_nightly_rates` por cada noche. Esos importes **no se modifican**
posteriormente: un cambio futuro en `rate_plans`/`seasonal_rates`/`daily_rate_overrides`
no altera reservas confirmadas. `rate_plan_id` en `reservation_nightly_rates`
usa `ON DELETE RESTRICT` para impedir el borrado de un plan referenciado por
el histórico.

### 10.3 `weekday_rates` y `weekdays` JSONB

Arrays de 7 valores lun-dom, almacenados como JSONB. La validación
(array de 7 números `>= 0`) se hace en CHECK con `jsonb_path_exists`
(inmutable, permitido en CHECK):

```sql
CHECK (
  jsonb_typeof(weekday_rates) = 'array'
  AND jsonb_array_length(weekday_rates) = 7
  AND NOT jsonb_path_exists(weekday_rates, '$[*] ? (@.type() != "number")')
  AND NOT jsonb_path_exists(weekday_rates, '$[*] ? (@.type() == "number" && @ < 0)')
)
```

> El backend debe mapear `weekday_rates` como `List<BigDecimal>` (7 elementos)
> con un `JsonAttributeConverter`. El orden es **lun-dom** (índice 0=lunes).

### 10.4 `UNIQUE NULLS NOT DISTINCT`

`daily_rate_overrides` y `closed_dates` usan `UNIQUE NULLS NOT DISTINCT`
(PostgreSQL 15+) para que las filas con `rate_plan_id IS NULL` /
`room_type_id IS NULL` (que significan "cualquier plan" / "cualquier tipo")
no admitan duplicados. Sin este modificador, PostgreSQL trataría los NULL
como distintos y permitiría múltiples filas `(1, NULL, '2026-12-31')`.

### 10.5 Impuestos y cargos (ER-5)

`taxes_and_fees` no se codifican como constantes (ER-5.5): IVA, ISH y
cualquier otro deben configurarse con vigencia (`valid_from`/`valid_to`).
Solo los `active = true` y vigentes en la fecha de la noche se aplican. El
snapshot de `reservation_nightly_rates` congela los importes calculados al
confirmar (ER-5.6): si un impuesto cambia vigencia, las reservas
confirmadas no se recalculan.

> Los valores semilla de V11 (IVA 16%, ISH 3%) son de **demostración** y no
> constituyen asesoría fiscal/legal.

## 11. Evolución de reservas (V6, ER-9 / ER-13.5 / ER-14)

### 11.1 Snapshot por noche (`reservation_nightly_rates`)

Ver §10.2. `included` puede pasar a `false` al reducir estancia (ER-13.3),
pero **nunca** a `true` de nuevo (regla de aplicación). El `total` de cada
fila se valida con CHECK aritmético:

```sql
CHECK (total = base_rate + extra_person_charge - discount_amount
             + taxes_amount + fees_amount)
```

`reservations.total_amount` = `SUM(reservation_nightly_rates.total) WHERE
included = true`; se recalcula solo al añadir/quitar noches.

### 11.2 Historial de modificaciones (`reservation_adjustments`)

Cada modificación (EXTEND, REDUCE, CHANGE_DATES, CHANGE_ROOM, CANCEL,
NO_SHOW, PENALTY) genera una fila con `old_value`/`new_value` (TEXT, p.ej.
fechas ISO o `room_id`), `reason`, `user_id` y `metadata` JSONB para volcado
estructurado opcional.

### 11.3 Reservas grupales (`reservation_groups`)

Un grupo tiene nombre, `contact_guest_id` (ON DELETE SET NULL), notas y
`created_by`. Cada reserva del grupo referencia `reservations.group_id`
(nullable, ON DELETE SET NULL). La cancelación grupal cancela todas las
reservas del grupo no terminadas (ER-14.2); la política de cancelación
aplica por reserva individual.

## 12. Políticas de cancelación y bloqueos (V7, ER-10 / ER-17)

### 12.1 `cancellation_policies`

`deadline_hours` = horas antes de `check_in` para cancelar sin penalización.
`penalty_type`: NONE, PERCENTAGE, FIXED, FIRST_NIGHT. `no_show_penalty_*`
específico para no-show (fallback a `penalty_*` si no se diferencia). Una
reserva referencia `cancellation_policy_id` al confirmar (copiada desde el
plan tarifario); el FK es ON DELETE SET NULL (borrar una política no
deshabilita reservas existentes).

### 12.2 `room_blocks` y EXCLUDE GiST

Patrón idéntico a `reservation_rooms`: columna `overlap_range` generada
STORED + `EXCLUDE USING gist (room_id WITH =, overlap_range WITH &&)`. La
exclusión opera sobre **todas** las filas, incluidas las `released_at NOT
NULL`: el backend debe `DELETE` o actualizar las filas liberadas antes de
crear un bloqueo solapante. Los conflictos entre `room_blocks` y
`reservation_rooms` no se previenen a nivel de BD (tablas separadas); el
backend valida al asignar habitación.

## 13. Housekeeping (V8, ER-18)

`rooms.housekeeping_status` (TEXT nullable, CHECK IN DIRTY/CLEANING/
INSPECTED/READY) es el estado de limpieza actual, **separado** de
`rooms.status` (estado operativo). Nullable para no romper filas
existentes (V1-V4 no tenían esta columna; el valor por defecto al añadir es
NULL).

`housekeeping_tasks` registra cada tarea. Flujo (ER-18.2):
`DIRTY` → (asignar) → `CLEANING` → (limpieza completa) → `INSPECTED` →
(inspección OK) → `READY` → (backend pone `rooms.status = AVAILABLE` y
`completed_at = now`). Si la inspección falla, el task vuelve a `DIRTY` con
nota (ER-18.4). Una habitación no puede pasar a `AVAILABLE` si su último
task no está `READY` (regla de aplicación).

## 14. Privacidad (V9, ER-25 / ER-26)

`privacy_requests` (EXPORT, RECTIFY, DELETE) con estado y responsable.
`personal_data_access_logs` audita cada lectura de datos no enmascarados
(action VIEW, EXPORT, MODIFY, ANONYMIZE). `guests.do_not_contact` (BOOLEAN
NOT NULL DEFAULT FALSE) marca exclusión de comunicaciones.

El CHECK `users_role_check` se amplió en V9 (DROP + ADD) para admitir
`HOUSEKEEPING`, `MANAGER`, `PRIVACY_OFFICER` además de los roles V1
(`ADMIN`, `RECEPCIONISTA`). V1 no se modifica: el ALTER se hace en V9.

La anonimización (ER-26.3) preserva `guest_id` y reservas/pagos (integridad
financiera); sustituye `first_name='ANONIMIZADO'`, `last_name=''`,
`email/phone/document_number/nationality=NULL`. Es irreversible.

## 15. Historial de ocupación — `room_stays` (V10, ER-16)

`room_stays` registra cada estancia de una habitación asociada a una
reserva. Permite el cambio de habitación durante estancia (ER-16.2): se
cierra el `room_stay` actual (`actual_check_out = now`) y se crea uno nuevo
para la habitación destino. `check_in`/`check_out` son DATE (planificadas);
`actual_check_in`/`actual_check_out` son TIMESTAMPTZ (instante real). Sin
EXCLUDE GiST: una misma habitación puede tener varios `room_stays`
consecutivos; el backend valida coherencia contra `reservation_rooms`.

## 16. Semilla del motor tarifario (V11)

Carga para desarrollo (ver `V11__seed_rate_engine.sql`):

- **4 `cancellation_policies`**: Flexible (72h, NONE), Semi-flexible (48h,
  FIRST_NIGHT), Strict (24h, PERCENTAGE 50), Non-refundable (0h, PERCENTAGE
  100, no-show PERCENTAGE 100).
- **4 `rate_plans`** (uno por `room_type`, `is_default = TRUE`):
  - `STD`  Standard → `room_type` 2 (Doble, base 1800) → 1800/noche todos los días.
  - `NRF`  Non-refundable → `room_type` 1 (Sencilla, base 1200) → 960 (x0.8).
  - `CORP` Corp → `room_type` 3 (Suite, base 4500) → 4050 (x0.9).
  - `WKND` Weekend → `room_type` 4 (Familiar, base 2800) → 2800 base, 3220 (x1.15) vie/sáb.
- **2 `seasonal_rates`** sobre `STD` (plan 1): Alta (2026-06-01 ..
  2026-09-01, x1.25), Baja (2026-09-01 .. 2026-12-01, x0.85).
- **2 `taxes_and_fees`**: IVA 16% e ISH 3% (TAX_PERCENT, ROOM_RATE,
  vigentes desde `CURRENT_DATE` sin fin). **Demostración**.
- **4 `daily_rate_overrides`**: Nochevieja 2026-12-31 a 2x base por tipo
  (`rate_plan_id` NULL = cualquier plan).
- **1 `promotion_rules`**: `EARLYBIRD10` (PERCENTAGE 10, `min_nights` 3,

- **Backfill de `reservation_nightly_rates`** para la reserva semilla #1
  (Doble, 2 noches): `base_rate` 1800.00, `taxes_amount` 342.00 (19% =
  IVA 16% + ISH 3%), `total` 2142.00/noche. `reservations.total_amount`
  actualizado a 4284.00 (2 × 2142.00). `cancellation_policy_id` = 2
  (Semi-flexible).

> El backfill usa el precio base del plan Standard (1800.00) **sin** aplicar
> la temporada Alta vigente en junio, para coincidir con el `nightly_price`
> histórico de la reserva semilla (V2/V4 = 1800.00). El snapshot queda
> inmutable (ER-9); una vez confirmada, la reserva no se recalcula.
